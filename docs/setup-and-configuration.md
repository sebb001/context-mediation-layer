# Setup and Configuration

## Requirements

- Node.js 22.5 or newer
- npm

The repository uses Node's built-in `node:sqlite` module, which currently
emits an `ExperimentalWarning` on Node 22. Treat the SQLite persistence layer
as acceptable for local evaluation and pilot work only, until a production
database and migration strategy are reviewed.

## Install, build, initialise

```sh
npm ci
npm run build
node dist/cli/index.js init --pretty
node dist/cli/index.js status --pretty
```

`init` creates `cml.config.json`, a local SQLite database under `./var`, a
local `./vault` directory, the default scope and the `Coordination` domain, a
default `local-operator` actor, a bootstrap root contract, a default agent
actor-type contract, and the CLI operator role binding.

## Configuration sources

Precedence:

1. CLI flags
2. Environment variables
3. `cml.config.json`

Files: `.env.example` (environment placeholders), `cml.config.example.json`
(transparent JSON example), `cml.config.json` (generated by `init`; keep out
of release workflows if it contains local paths).

| Purpose | CLI flag | Environment | Config path |
| --- | --- | --- | --- |
| Config file | `--config` | `CML_CONFIG` | n/a |
| SQLite DB | `--db` | `CML_DB_PATH` | `database.path` |
| Actor name | `--actor` | `CML_ACTOR` | `actor.name` |
| Actor id | `--actor-id` | `CML_ACTOR_ID` | `actor.id` |
| Scope | `--scope` | `CML_SCOPE` | `defaultScope` |
| Vault root | `--vault-root` | `CML_VAULT_ROOT` | `vault.root` |

### Secrets

Use placeholders in committed files; real values belong in local environment
files or a secret manager: `CML_MCP_TOKEN`, `CML_PUBLIC_MCP_TOKEN`,
`CML_PUBLIC_MCP_UPSTREAM_TOKEN`, and any provider API keys used by runtime
adapters.

## MCP setup

Generate transparent client/server snippets:

```sh
node dist/cli/index.js setup mcp --transport stdio --out ./mcp.stdio.json
node dist/cli/index.js setup mcp --transport http --out ./mcp.http.json
node dist/cli/index.js setup mcp --transport public --out ./mcp.public.json
```

The generated files contain token placeholders and can be adapted to the
target client or deployment surface.

### Code agents (local)

Code agents with shell access should use the CLI or the project-scoped stdio
MCP server:

```sh
npm run build
node dist/cli/index.js setup mcp --transport stdio --out ./.mcp.json
claude mcp list
```

The generated `.mcp.json` exposes `cml-mcp` with local database and actor
environment variables. Clients prompt before using project-scoped MCP servers;
approve only in trusted working copies.

### Chat clients (remote)

Chat clients that cannot run the local repo need the hosted public MCP
endpoint, configured with the URL ending in `/mcp`:

```text
https://your-public-mcp.example/mcp
```

When OAuth is enabled, use the client's remote-connector OAuth flow.

## Public gateway

The public MCP gateway expects: the upstream URL for the local HTTP MCP
server; an upstream bearer token when the upstream is not local; a bearer
token for public requests or OAuth/OIDC issuer settings; a public base URL for
OAuth discovery; allowlisted tool names; and optional required-actor and
writable-path rules.

Keep gateway configs narrow for pilots. Start read-only, then add write tools
only after the mandate, actor mapping, path policy, and audit path are
reviewed. Do not expose production data, regulated production workflows, or
autonomous downstream writes through the public gateway; the pilot boundary in
[Concepts and Limitations](concepts-and-limitations.md) applies.

## OAuth/OIDC

The gateway verifies RS256 OAuth/OIDC JWT bearer tokens directly. Set
`CML_OAUTH_ISSUER` to verify tokens from an external issuer, or
`CML_OAUTH_PILOT_ISSUER=1` to use the built-in pilot issuer. Both modes expose
the universal MCP authorisation surface
(`/.well-known/oauth-protected-resource`, `WWW-Authenticate` with
`resource_metadata`); the pilot issuer additionally exposes the authorisation
server endpoints (`/authorize`, `/token`, `/register`, `/jwks.json`, and the
well-known discovery documents).

| Purpose | Environment |
| --- | --- |
| Public MCP base URL | `CML_PUBLIC_MCP_BASE_URL` |
| Upstream MCP URL | `CML_PUBLIC_MCP_UPSTREAM_URL` |
| Upstream MCP bearer token | `CML_PUBLIC_MCP_UPSTREAM_TOKEN` |
| OIDC issuer | `CML_OAUTH_ISSUER` |
| Protected MCP resource | `CML_OAUTH_RESOURCE` |
| Expected audience | `CML_OAUTH_AUDIENCE` |
| Supported scopes | `CML_OAUTH_SCOPES` |
| JWKS override | `CML_OAUTH_JWKS_URL` |
| Subject-to-actor map | `CML_OAUTH_SUBJECT_ACTOR_MAP` |
| Group-to-actor map | `CML_OAUTH_GROUP_ACTOR_MAP` |
| Read-only fallback actor | `CML_OAUTH_DEFAULT_ACTOR` |
| Built-in pilot issuer | `CML_OAUTH_PILOT_ISSUER=1` |
| Pilot RSA private key | `CML_OAUTH_PRIVATE_KEY_PATH` |
| Pilot key id | `CML_OAUTH_KEY_ID` |
| Pilot client/refresh store | `CML_OAUTH_CLIENT_STORE_PATH` |
| Pilot browser access key | `CML_OAUTH_AUTH_SECRET` |
| Pilot subject | `CML_OAUTH_PILOT_SUBJECT` |
| Pilot groups | `CML_OAUTH_PILOT_GROUPS` |

Actor maps may be JSON objects or comma-separated `claim=actor` pairs.

Bootstrap local pilot key material without committing secrets:

```sh
npm run oauth:bootstrap -- --base-url https://your-public-mcp.example --actor pilot-user
```

This writes ignored local files under `var/oauth/` and `.env.oauth.local`. The
generated access key is used only on the browser authorisation screen; keep it
in a secret manager for any shared pilot.

When OAuth is enabled: insecure mode is rejected at startup; client-supplied
`actor` values cannot override the mapped actor; unmapped tokens are rejected
unless a default actor is configured; and non-local upstream MCP servers
require `CML_PUBLIC_MCP_UPSTREAM_TOKEN`.

OAuth provides secure MCP access, scoped write permissions, and actor mapping.
It does not by itself provision durable end-user identities: for end-to-end
identity, configure an external issuer and map `sub`, group, or role claims to
explicit CML actors. The built-in pilot issuer stores authorisation codes in
memory and registered clients/refresh tokens in
`CML_OAUTH_CLIENT_STORE_PATH` when configured — a controlled pilot issuer,
not production identity infrastructure.

## Contract imports

Import a lightweight contract pack:

```sh
node dist/cli/index.js contract import --file ./contracts.example.json --pretty
```

The JSON object may include `contracts`, `actorTypes`, and `roles` arrays. The
CLI writes through the service layer and attributes imports to the configured
actor. Active contract behaviour lives in the registry after import, not in the
JSON file that seeded it.
